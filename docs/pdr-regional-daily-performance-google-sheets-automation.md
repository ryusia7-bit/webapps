## 1. 문제 정의

### 현재 상황

정신건강팀은 매일 담당자별 실적을 엑셀 파일의 월별 시트에 수기 입력하고 있으며, 법인월보고·분기보고 등 집계 시트는 해당 데이터를 수식으로 참조하는 구조다. 그러나 이 구조는 아래와 같은 복합적 문제를 안고 있다.

### 핵심 문제

- **이중 입력 부담:** 일일실적 시트에 집계값을 입력하는 동시에, 입원현황·상담현황·연계대장 등 개별 DB 시트에도 동일 사건을 케이스별로 별도 입력해야 한다.
- **수식 파손 반복:** 연도·월 교체 시 시트를 수동 복사하면서 `#REF!` 오류가 대량 발생한다. 법인월보고용, 분기보고용, 연계대장 등 다수 시트에서 확인되었다.
- **입력 오류 위험:** 담당자가 자신의 행을 직접 찾아 값을 넣는 구조여서 엉뚱한 셀 입력, 날짜 형식 불일치(`7/12` vs `2023-7-12`) 등의 오류가 빈번하다.
- **집계 시트의 불완전 연동:** 법인월보고용과 분기보고용이 월별 시트를 수식으로 참조하지만, 시트 이름이 달라지면 참조가 끊어진다.

## 2. 목표 및 성공 기준

### 목표

> **DB 데이터(케이스별 상세 기록)를 1회 입력하면, 구글시트 내 각 제목별 시트에 집계 실적이 자동으로 반영된다.**

### 성공 기준

| 기준 | 세부 내용 |
| --- | --- |
| 단일 입력 원칙 | 담당자는 DB 시트에 1회만 입력한다. 집계 시트에는 별도 입력이 필요 없다. |
| 오류 제거 | 날짜 형식 오류, `#REF!` 등 수식 파손이 발생하지 않는다. |
| 시트 자동 반영 | 입력 즉시 법인월보고용·분기보고용·일일실적 등 집계 시트가 갱신된다. |
| 유지보수 독립성 | 연도·월이 바뀌어도 시트 복사·수식 수정 없이 운영된다. |

## 3. 현재 업무 흐름과 비효율

### 현재 흐름 (As-Is)

```text
[현장 활동]
    ↓
[담당자] 월별 시트(2026년1월 등)에서
         자신의 행을 직접 찾아 날짜별 집계값 수기 입력
    ↓
[동시에] 입원현황 / 상담현황 / 연계대장 / 시설입소현황 등
         개별 DB 시트에도 케이스별 상세 데이터 별도 입력
    ↓
[집계 시트] 법인월보고용·분기보고용이 월별 시트를 수식으로 참조
            → 시트명 변경 시 수식 파손(#REF!)
```

### 주요 비효율 포인트

- 동일 사건을 최소 2개 시트에 이중 입력한다. 예: 자의입원 1건 → 일일실적 시트 + 입원현황 시트
- 월별 시트에 하루당 약 10행(담당자 8명 + 일일누계 + 총누계)이 고정 배열되어 있어, 담당자 추가·변경 시 전체 구조 재편이 필요하다.
- 날짜 형식 주의사항이 별도 안내 문구로 존재할 정도로 오입력 위험이 높다.
- 법인월보고용·분기보고용·시설입소실적·입원실적 시트 등 다수 집계 시트에서 현재도 `#REF!` 오류가 발견된다.

### 4.2 월별 일일실적 시트 행 구조

```text
행 1 (헤더) : 날짜 | 담당 | 상담(거리 및 내방) [정신/알콜/기타/소계] | 조치 [자의입원/응급입원/보호진단/시설입소(노숙)/시설입소(타복지)/지원주택/주거지원/물품제공/기타] | 조치후사례관리 [외래진료/병원방문/주거지방문/투약관리/기타] | 합계
행 2 (날짜) : 조국일 행
행 3        : 최리선 행
...
행 9        : 마명철 행 (담당자 8명)
행 10       : 일일누계 행 (해당일 합계)
행 11       : 총누계 행 (월 누적 합계)
행 12 (다음날): 조국일 행
...
```

- 하루당 10행 반복 구조 (담당자 8명 + 일일누계 + 총누계)
- 월별 시트 크기: 약 305~338행 × 40열 (31일 × 10행 + 헤더 2행 + 주차별 소계)

## 5. DB 입력 구조 및 필드 정의

### 5.1 자동화 설계 방향

현재 DB 시트들(입원현황, 상담현황 등)이 이미 케이스별 원천 데이터를 보유하고 있다. 이를 **단일 입력 소스(Single Source of Truth)**로 삼아, 집계 시트가 이 DB를 직접 `QUERY`·`SUMIF`·`COUNTIFS`로 참조하도록 재설계한다.

월별 일일실적 시트(집계 입력 시트)는 **집계 결과를 표시하는 뷰(View)** 역할로 전환된다.

### 5.2 주요 DB 시트 및 핵심 필드 정의

#### [DB-01] 일일활동 DB

현재 월별 일일실적 시트를 대체하는 새로운 DB다.

| 필드명 | 타입 | 설명 | 예시 |
| --- | --- | --- | --- |
| 날짜 | DATE | 활동 날짜 | 2026-01-02 |
| 담당자 | TEXT | 담당 사회복지사 이름 | 조국일 |
| 항목구분 | TEXT | 실적 항목 대분류 | 상담, 조치, 사례관리 |
| 세부항목 | TEXT | 항목 세분류 | 정신상담, 자의입원, 외래진료 |
| 건수 | NUMBER | 해당 날짜/담당자/항목의 건수 | 3 |

> [가정] 현재 월별 시트의 입력 단위(날짜×담당자×항목)는 이미 집계된 값(건수)이며, 원시 케이스 기록은 별도 DB 시트(입원현황, 상담현황 등)에 존재한다. 두 방식 중 어느 것을 DB로 삼을지 아래에서 구분한다.

### 5.3 두 가지 DB 입력 방식 비교

#### 방식 A: 집계값 입력 방식 (현행 유지형)

- 담당자가 날짜×항목 단위로 건수를 입력하는 `일일활동DB` 시트를 별도 운영한다.
- 기존 월별 시트를 이 DB 시트로 대체한다.
- 집계 시트는 이 DB를 `SUMIFS`로 집계한다.

#### 방식 B: 케이스 기록 기반 자동 집계 방식 (완전 자동화형)

- 입원현황, 상담현황, 시설입소현황 등 케이스 DB에 입력하면
- Apps Script가 날짜·담당자·항목을 자동 분류하여 집계 시트에 반영한다.

> [추가 확인 필요] 방식 A와 B 중 어느 쪽을 선택할지는 팀의 입력 습관과 운영 우선순위에 따라 결정해야 한다. 두 방식 혼용도 가능하다. 예: 상담·입원은 케이스DB 자동집계, 기타 항목은 집계값 직접 입력

아래 설계는 **방식 A(집계값 입력 DB)를 기본으로, 방식 B와의 연동을 부분 구현하는 혼합 방식**을 전제로 한다.

## 6. 구글시트 반영 구조

### 6.1 전체 시트 구성

자동화 후 구글시트 파일은 아래 3계층으로 구성된다.

```text
[계층 1: DB 시트 (입력 계층)]
├── 일일활동DB         ← 담당자 일일 집계값 입력 (신규)
├── 상담현황DB         ← 케이스별 상담 기록 (기존 상담현황(2026) 시트 활용)
├── 입원현황DB         ← 케이스별 입원 기록 (기존 입원현황 시트 활용)
├── 시설입소현황DB     ← 케이스별 시설 입소 기록 (기존 시트 활용)
├── 연계대장DB         ← 케이스별 연계 기록 (기존 시트 활용)
├── 조치후사례관리DB   ← 케이스별 사례관리 기록 (기존 시트 활용)
└── (기타 DB 시트들)

[계층 2: 집계 시트 (자동 반영 계층)]
├── 일일실적(월별뷰)   ← 일일활동DB를 월별로 집계하여 자동 표시
├── 법인월보고용       ← DB들을 월/분기/연간 집계하여 자동 표시
├── 분기보고용         ← DB들을 분기 단위로 집계하여 자동 표시
├── 입원실적           ← 입원현황DB 집계
├── 시설입소실적       ← 시설입소현황DB 집계
├── 연계실적           ← 연계대장DB 집계
└── 조치후사례관리(집계) ← 조치후사례관리DB 집계

[계층 3: 참조 시트 (설정 계층)]
├── 기준(담당자목록)   ← 담당자, 항목 등 유효성 검사 기준값
└── 도움말             ← 입력 가이드
```

### 6.2 데이터 흐름도

```text
담당자 → [일일활동DB] 날짜·담당자·항목별 건수 입력
                ↓ SUMIFS / QUERY
        [일일실적(월별뷰)]  ← 월별 달력 형태로 자동 집계 표시
                ↓ SUMIFS
        [법인월보고용]       ← 월/분기/연간 집계 자동 표시
                ↓ SUMIFS
        [분기보고용]         ← 분기별 항목 집계 자동 표시

케이스 기록 담당자 → [입원현황DB] 케이스별 입원 입력
                            ↓ COUNTIFS / SUMIFS
                    [입원실적]     ← 월별 입원 유형·성별·퇴원조치 집계

케이스 기록 담당자 → [시설입소현황DB] 케이스별 입소 입력
                              ↓ COUNTIFS
                    [시설입소실적]     ← 월/분기 시설 유형별 집계

케이스 기록 담당자 → [연계대장DB] 케이스별 연계 입력
                          ↓ COUNTIFS
                  [연계실적]      ← 월/주별, 내방경위별 집계
```

## 7. 시트 제목별 매핑 방식

### 7.1 법인월보고용 시트

| 집계 셀 | 참조 DB | 집계 함수 | 조건 |
| --- | --- | --- | --- |
| 1월 정신상담 | 일일활동DB | `SUMIFS(정신상담열, 날짜열, ">=2026-01-01", 날짜열, "<=2026-01-31")` | 년월 범위 |
| 1월 알코올상담 | 일일활동DB | `SUMIFS(알콜상담열, 날짜열, 1월범위)` | 년월 범위 |
| 1월 자의입원 | 입원현황DB | `COUNTIFS(월열, 1, 입원형태열, "자의입원")` | 월 + 유형 |
| 1월 시설입소 | 시설입소현황DB | `COUNTIFS(월열, 1)` | 월 |
| 1월 외래진료 | 일일활동DB | `SUMIFS(외래진료열, 날짜열, 1월범위)` | 년월 범위 |
| 분기합 | 위 집계값 | `SUM(1월셀:3월셀)` | 분기 합산 |

> [가정] 법인월보고용의 `수급진행`, `주민등록복원` 항목은 조치후사례관리DB에서 집계한다. 현재 해당 DB 시트에 월별 집계 행이 이미 있으므로 해당 값을 참조한다.

### 7.2 분기보고용 시트

| 섹션 | 참조 DB | 집계 방식 |
| --- | --- | --- |
| 1. 상담 (정신과/알콜/기타) | 일일활동DB | 분기 내 날짜 범위 `SUMIFS` |
| 2. 조치 - 병원입원 | 입원현황DB | `COUNTIFS(입원형태별, 분기 월 범위)` |
| 2. 조치 - 시설입소 | 시설입소현황DB | `COUNTIFS(종류별, 분기 월 범위)` |
| 2. 조치 - 주거지원 | 일일활동DB 또는 시설입소현황DB | `SUMIFS` |
| 3. 사례관리 | 조치후사례관리DB | `COUNTIFS(지원유형별, 분기 월 범위)` |

> [추가 확인 필요] 분기보고용 시트의 `직접발굴·센터거리상담반·위기대응콜·노숙인시설개입요청` 등 연계경로별 항목은 연계대장DB에서 집계하는 것으로 설계하였으나, 현재 일일실적 시트에 해당 항목이 없어 DB 간 매핑 확인이 필요하다.

### 7.3 일일실적(월별뷰) 시트

기존 월별 시트의 시각적 레이아웃(날짜별, 담당자별 행 구조)을 유지하되, 셀 값은 직접 입력이 아닌 **일일활동DB를 `QUERY`/`SUMIFS`로 참조**하여 자동 표시한다.

```gs
=SUMIFS(일일활동DB!C:C,
        일일활동DB!A:A, "2026-01-02",
        일일활동DB!B:B, "조국일")
```

예: `C5` 셀 (1월 2일, 조국일의 정신상담 건수)

> [가정] 월별뷰 시트는 읽기 전용으로 운영되며, 담당자는 `일일활동DB` 시트에서만 입력한다.

### 7.4 입원실적 시트

```gs
=COUNTIFS(입원현황DB!입원형태열, "자의입원", 입원현황DB!월열, 1)
```

예: 1월 자의입원 건수

```gs
=COUNTIFS(입원현황DB!월열, 1, 입원현황DB!성별열, "남")
```

예: 1월 남성 입원 건수

### 7.5 연계실적 시트

```gs
=COUNTIFS(연계대장DB!월열, 1, 연계대장DB!주열, 1, 연계대장DB!성별열, "남")
```

예: 1월 1주차 남성 연계 건수

```gs
=COUNTIFS(연계대장DB!월열, 1, 연계대장DB!내방경위열, "노숙인방문")
```

예: 1월 노숙인방문 경로 건수

## 8. 자동화 처리 흐름

### 8.1 Google Apps Script 활용 방안

구글시트 수식(`SUMIFS`, `COUNTIFS`, `QUERY`)만으로도 대부분의 자동화가 가능하나, 아래 기능은 Apps Script를 활용한다.

#### ① 입력 시 자동 타임스탬프 및 입력자 기록

```text
트리거: onEdit(e)
동작: 일일활동DB 시트에 새 행이 추가되면
      → T열(입력자): Session.getActiveUser().getEmail()
      → U열(입력일시): new Date()
      자동 기록
```

#### ② 날짜 유효성 자동 보정

```text
트리거: onEdit(e)
동작: 날짜열(A열)에 텍스트 형식으로 입력 감지 시
      → 자동으로 DATE 형식으로 변환
      → 경고 메시지 표시
```

#### ③ 월별뷰 시트 자동 생성 (선택적)

```text
트리거: 수동 실행 또는 월 첫째 날 시간 기반 트리거
동작: 신규 월 시작 시 월별뷰 시트 자동 생성
      → 해당 월의 날짜·담당자 행 구조 자동 세팅
      → 각 셀에 SUMIFS 수식 자동 삽입
```

#### ④ 보고서 자동 이메일 발송 (선택적)

```text
트리거: 매월 말일 시간 기반 트리거
동작: 법인월보고용 시트의 당월 데이터를 PDF로 변환
      → 지정 이메일 주소로 자동 발송
```

### 8.2 처리 흐름 단계

#### Step 1: 일일 입력

1. 담당자가 `일일활동DB` 시트에 접속한다.
2. 날짜 선택기에서 날짜를 선택한다. 자동 형식이 적용된다.
3. 담당자 드롭다운에서 본인을 선택한다.
4. 해당 날짜 활동 건수를 입력한다. 빈 칸은 0으로 처리한다.
5. Apps Script가 입력자·입력일시를 자동 기록한다.

#### Step 2: 자동 집계 반영

1. 입력 즉시 `SUMIFS`/`COUNTIFS` 수식이 갱신된다.
2. `일일실적(월별뷰)` 시트에서 해당 날짜·담당자 셀이 자동 업데이트된다.
3. `법인월보고용`, `분기보고용`, `입원실적` 등 집계 시트가 동시 갱신된다.

#### Step 3: 케이스 DB 병행 관리

1. 입원, 시설입소, 연계 등 케이스는 각 DB 시트에 별도 입력한다.
2. `입원실적`, `시설입소실적`, `연계실적` 시트는 해당 DB를 `COUNTIFS`로 자동 집계한다.

## 9. 예외 케이스 및 검증 방안

### 9.1 예상 예외 케이스

| 예외 상황 | 발생 조건 | 처리 방안 |
| --- | --- | --- |
| 날짜 오입력 | 텍스트 형식으로 날짜 입력 | 데이터 유효성 검사로 날짜 형식 강제; Apps Script로 감지·경고 |
| 담당자 이름 불일치 | 드롭다운 외 직접 입력 | 드롭다운 강제(목록 외 입력 차단) |
| 동일 날짜·담당자 중복 행 입력 | 실수로 같은 날짜 2행 입력 | `SUMIFS`는 자동 합산; 단, 조건부 서식으로 중복 행 시각 경고 |
| 음수 입력 | 건수에 음수 입력 | 데이터 유효성 검사로 0 이상 정수만 허용 |
| 담당자 추가·퇴직 | 기준 시트의 담당자 목록 변경 필요 | `기준` 시트의 담당자 목록만 수정하면 드롭다운·월별뷰 자동 반영 |
| 연도 변경 | 2027년 1월 시작 | 날짜 기반 집계이므로 시트 복사 불필요; `법인월보고용`의 연도 파라미터만 변경 |
| 공휴일·휴무일 | 활동 건수가 0인 날 | 행 미입력 또는 0 입력 모두 허용; 집계 시 0으로 처리 |

### 9.2 데이터 검증 방안

#### 입력 단계 검증

- 날짜열: 구글시트 데이터 유효성 → 날짜만 허용
- 담당자열: 드롭다운 목록 외 입력 차단
- 건수열: 데이터 유효성 → 0 이상의 정수만 허용

#### 집계 단계 검증

- 월별뷰의 담당자별 합계와 `일일활동DB`의 담당자별 `SUMIFS`가 일치하는지 검증 행을 추가한다.
- `법인월보고용`의 집계값과 케이스DB의 `COUNTIFS`가 일치하는지 비교 열을 추가한다. `차이 = 0`이어야 정상이다.

#### 정기 검증

- 매월 말 `입원실적` 시트의 자의·응급·보호진단 합계와 `입원현황DB`의 `COUNTA` 값을 비교한다.
- 불일치 발생 시 조건부 서식으로 해당 셀을 빨간색으로 강조한다.

## 10. 운영 방식 및 유지보수 고려사항

### 10.1 구현 우선순위

| 단계 | 작업 내용 | 소요 예상 | 우선순위 |
| --- | --- | --- | --- |
| 1단계 | `일일활동DB` 시트 생성 및 유효성 검사 설정 | 1~2일 | ★★★ |
| 2단계 | `법인월보고용` `SUMIFS` 수식 재구성 (`#REF!` 해소) | 1~2일 | ★★★ |
| 3단계 | `분기보고용`·`입원실적`·`연계실적` 수식 재구성 | 2~3일 | ★★★ |
| 4단계 | 월별뷰 시트를 `SUMIFS` 자동참조 구조로 전환 | 1~2일 | ★★ |
| 5단계 | Apps Script 타임스탬프·유효성 보조 기능 | 1일 | ★★ |
| 6단계 | 월별뷰 자동생성·이메일 발송 스크립트 | 2~3일 | ★ |

### 10.2 운영 규칙

- **입력 원칙:** 담당자는 `일일활동DB` 시트에만 입력하며, 집계 시트 직접 수정은 금지한다.
- **입력 주기:** 당일 활동을 당일 또는 익일 오전 중 입력하도록 권장한다.
- **DB 시트 분리:** DB 시트는 탭 색상으로 구분한다. DB: 파란색, 집계: 초록색, 설정: 회색
- **수식 보호:** 집계 시트의 수식 셀에 시트 보호를 설정해 실수 수정을 방지한다.

### 10.3 연도 전환 절차 (연간 1회)

현행 구조의 가장 큰 비효율인 연도 전환을 아래와 같이 단순화한다.

```text
기존: 새 월 시트 복사 → 날짜 수동 수정 → 수식 재연결 → #REF! 수정
개선: 법인월보고용의 연도 파라미터 셀(예: B1 = 2026) 값을 2027로 변경
      → 전체 SUMIFS의 날짜 범위 자동 갱신 (연도 1개 셀만 수정)
```

> [가정] `법인월보고용`의 날짜 범위 수식을 `DATE($B$1, 월, 1)` 형태의 동적 수식으로 구성하는 것을 전제로 한다.

### 10.4 접근 권한 관리

| 시트 유형 | 편집 권한 | 보기 권한 |
| --- | --- | --- |
| DB 시트 (`일일활동DB` 등) | 팀원 전체 | 팀원 전체 |
| 집계·보고 시트 | 관리자만 (수식 보호) | 팀원 전체 |
| 기준(설정) 시트 | 관리자만 | 팀원 전체 |

> [추가 확인 필요] 구글 계정 및 구글 워크스페이스 사용 여부 확인이 필요하다. 개인 Gmail 계정 환경에서는 Apps Script 공유 실행 시 권한 정책이 다를 수 있다.

### 10.5 유지보수 시나리오

| 상황 | 조치 방법 |
| --- | --- |
| 담당자 추가 | `기준` 시트의 담당자 목록에 추가 → 드롭다운 자동 반영 |
| 실적 항목 추가 | `일일활동DB`에 열 추가 → 집계 시트에 `SUMIFS` 열 추가 |
| 보고 양식 변경 | 집계 시트의 레이아웃 조정 (수식 범위 수정) |
| 과거 데이터 소급 입력 | `일일활동DB`에 과거 날짜로 행 추가 → 집계 자동 반영 |

## 부록: 가정·전제·미확인 사항 요약

### [가정] 항목

- 담당자 8명(조국일, 최리선, 안영호, 박상준, 오강현, 김영민, 임종혁, 마명철)은 2026년 기준이며 변동 가능성이 있다.
- `법인월보고용`의 `수급진행`, `주민등록복원` 항목은 `조치후사례관리DB`에서 집계한다.
- 집계값 입력 방식(방식 A)을 우선 채택하되, 케이스DB와의 교차 검증 구조를 병행 구현한다.
- 구글시트 `SUMIFS`/`QUERY` 수식은 데이터 범위가 열 전체(예: `A:A`)를 참조하는 방식으로 확장성을 확보한다.

### [제약] 항목

- 구글시트는 동시 편집 시 수식 계산 충돌이 발생할 수 있으며, 대규모 `SUMIFS` 수식은 10만 행 이상에서 성능 저하 가능성이 있다.
- Apps Script의 시간 기반 트리거(매월 보고서 발송 등)는 구글 계정 유형(개인/워크스페이스)에 따라 실행 제한이 다를 수 있다.

### [추가 확인 필요] 항목

- 방식 A(집계값 입력)와 방식 B(케이스기록 기반 자동집계) 중 최종 채택 방향 결정이 필요하다.
- `분기보고용` 시트의 연계경로별 항목이 `연계대장DB`와 정확히 매핑되는지 현장 확인이 필요하다.
- 구글 워크스페이스 계정 사용 여부 및 Apps Script 실행 권한 수준 확인이 필요하다.
- `상담현황(2026)` 시트가 현재 거의 비어 있는 이유 확인이 필요하다. 2026년 데이터 미입력 상태인지, 별도 관리 중인지 확인해야 한다.
- `법인월보고용`의 `기초상담` 항목 정의 및 집계 소스 확인이 필요하다. 현재 일일실적 시트에 해당 항목이 없다.
