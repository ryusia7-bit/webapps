# 현장용 정신건강 척도검사 웹앱 PRD

문서 상태: Draft v1  
작성일: 2026-03-24  
대상 프로젝트: `scale-screening-web-app`

## 1. 제품 요약

정신건강 사회복지사가 상담 현장, 방문 현장, 기관 내 면담 공간에서 빠르게 척도검사를 수행하고 결과를 즉시 확인하며, 검사 기록을 구글 시트에 누적 저장할 수 있는 모바일 우선 웹앱을 구축한다.

이 제품의 핵심은 `현장에서 검사하기 쉬운 UI`와 `운영팀이 누적 데이터를 관리하기 쉬운 저장 구조`를 동시에 만족시키는 것이다.

## 2. 배경과 문제 정의

현재 척도검사는 기관 홈페이지 또는 종이 문항지, 개별 엑셀, 구글 폼, 수기 기록 등으로 분산 수행되기 쉽다. 이 방식은 아래 문제를 만든다.

- 현장에서 척도 문항을 찾고 실행하는 데 시간이 걸린다.
- 점수 계산과 결과 정리가 수작업에 의존한다.
- 사회복지사마다 기록 방식이 달라 데이터 정리가 어렵다.
- 검사 결과가 기관 차원의 누적 데이터로 정리되지 않는다.
- 모바일 환경에서 사용성이 낮아 대상자 앞에서 진행이 매끄럽지 않다.

## 3. 제품 목표

### 3.1 핵심 목표

- 정신건강 사회복지사가 1분 이내에 원하는 척도를 열 수 있어야 한다.
- 검사 완료 즉시 점수와 결과 요약이 자동 계산되어야 한다.
- 검사 결과를 로컬에 임시 저장하고 필요 시 구글 시트로 전송할 수 있어야 한다.
- 모바일과 태블릿에서 버튼, 글자, 입력 구조가 현장 사용에 적합해야 한다.
- 척도 구조를 데이터 기반으로 관리하여 신규 척도 추가가 쉬워야 한다.

### 3.2 비목표

- 의료 진단 시스템 구축
- EMR 연동
- 민감한 주민등록번호, 상세 생년월일, 상세 주소 저장
- 고도 통계 대시보드 구현
- 외부 공개 서비스 운영

## 4. 제품 방향

본 제품은 `현장 검사 수행 도구`로서의 웹앱을 1차 제품으로 삼는다.  
구글 시트 또는 AppSheet는 `기록 관리와 조회 도구`로 보조적으로 활용한다.

즉, 역할 분담은 다음과 같다.

- 웹앱: 척도 선택, 문항 응답, 점수 계산, 결과 확인, 현장 저장, 전송
- 구글 시트: 누적 기록 저장, 문항 응답 보관, 척도 마스터 관리
- AppSheet: 필요 시 관리자 조회 화면 또는 간단한 기록 열람 도구

## 5. 대상 사용자

### 5.1 1차 사용자

- 정신건강 사회복지사
- 자살예방 및 정신건강 상담 현장 실무자
- 사례관리, 방문상담, 초기상담 담당자

### 5.2 2차 사용자

- 팀장 또는 관리자
- 데이터 취합 담당자
- 사업 실적 및 모니터링 담당자

## 6. 대표 사용 시나리오

### 6.1 현장 면담 시나리오

사회복지사가 대상자와 면담 중 태블릿 또는 휴대폰에서 척도 앱을 열고, PHQ-9 또는 GAD-7 같은 척도를 선택해 즉시 검사를 진행한다. 응답 완료 후 점수와 위험도 요약을 바로 확인하고, 상담 메모를 남긴 뒤 기록을 저장한다.

### 6.2 방문상담 시나리오

인터넷이 불안정한 환경에서도 실무자는 척도를 불러와 검사하고, 우선 브라우저에 저장한 뒤 네트워크가 안정적인 시점에 구글 시트로 전송한다.

### 6.3 팀 운영 시나리오

관리자는 구글 시트에서 검사 기록, 문항 응답, 척도 마스터를 확인하고 추후 AppSheet나 피벗 테이블로 누적 현황을 조회한다.

## 7. MVP 범위

### 7.1 포함 범위

- 8개 척도 제공
- 척도 목록 선택 UI
- 응답자 기본 정보 입력
- 문항 응답 UI
- 자동 점수 계산 및 결과 요약
- 로컬 저장
- JSON/CSV 내보내기
- 인쇄용 결과지
- 구글 시트 전송
- 척도 마스터 전송

### 7.2 제외 범위

- 사용자 계정 시스템
- 대상자 장기 프로필 관리
- 기관 전체 통계 대시보드
- 결재/승인 워크플로우
- 푸시 알림

## 8. 제공 척도 범위

MVP 기준 대상 척도는 아래 8개다.

- 우울 `PHQ-9`
- 불안 `GAD-7`
- 조기정신증 `mKPQ-16`
- 조울증 `K-MDQ`
- 스트레스 `PSS`
- 불면 `ISI-K`
- 알코올중독 `AUDIT-K`
- 사건충격척도 `IES-R`

향후 척도 추가는 코드 수정이 아닌 데이터 추가 중심으로 처리하는 것을 원칙으로 한다.

## 9. 기능 요구사항

### FR-01 척도 선택

- 사용자는 앱 진입 후 척도 목록을 즉시 볼 수 있어야 한다.
- 각 척도는 이름, 약어, 권장 연령, 문항 수를 표시해야 한다.
- 척도 목록은 모바일 화면에서 스크롤과 터치에 최적화되어야 한다.

### FR-02 응답자 기본 정보

- 검사일은 필수 입력이다.
- 담당자명, 대상자 식별 메모, 현장 메모를 입력할 수 있어야 한다.
- 척도별 응답자 정보 문항이 있으면 렌더링되어야 한다.
- 개인정보 최소수집 원칙을 유지해야 한다.

### FR-03 문항 응답

- 각 문항은 보기 선택 방식으로 쉽게 응답할 수 있어야 한다.
- 필수 응답 누락 시 어느 항목이 비었는지 안내해야 한다.
- 진행률을 확인할 수 있어야 한다.
- 긴 문항도 모바일에서 가독성 있게 보여야 한다.

### FR-04 점수 계산과 결과 제공

- 척도별 점수 계산 규칙을 자동 적용해야 한다.
- 총점, 요약 결과, 필요 시 문항별 응답 내역을 보여줘야 한다.
- 결과는 현장 설명용으로 간결해야 하며, 진단 도구가 아니라는 안내를 유지해야 한다.

### FR-05 저장과 재열람

- 사용자는 결과를 브라우저 로컬 저장소에 저장할 수 있어야 한다.
- 저장된 결과는 다시 열람, 인쇄, JSON 내보내기가 가능해야 한다.
- 최근 기록은 기본적으로 50건 이내를 보관한다.

### FR-06 구글 시트 동기화

- 사용자는 Apps Script Web App URL만 입력해도 전송할 수 있어야 한다.
- 현재 결과 1건 전송과 저장 기록 전체 전송을 모두 지원해야 한다.
- 척도 마스터도 별도 전송할 수 있어야 한다.
- 전송 대상 시트는 아래 탭 구조를 사용한다.
  - `척도검사기록`
  - `척도문항응답`
  - `척도마스터`
  - `척도문항마스터`
  - `척도선택지마스터`

### FR-07 인쇄와 내보내기

- 검사 결과를 인쇄용 레이아웃으로 확인할 수 있어야 한다.
- 저장 기록을 CSV와 JSON으로 내보낼 수 있어야 한다.
- 기관명, 팀명, 연락 메모를 결과지 상단에 반영할 수 있어야 한다.

### FR-08 오류 처리

- URL 누락, 네트워크 오류, 전송 실패 시 사용자에게 이해하기 쉬운 메시지를 제공해야 한다.
- 척도 로딩 실패 시 새로고침 안내만 하지 않고 대체 경로를 제공해야 한다.
- 동기화 성공 여부를 시트 반영 기준으로 확인할 수 있는 운영 가이드가 필요하다.

## 10. 비기능 요구사항

### 10.1 사용성

- 모바일 우선 설계
- 태블릿 가로/세로 대응
- 큰 버튼, 충분한 줄간격, 명확한 선택 상태 표시
- 50대 이상 실무자도 큰 학습 없이 사용할 수 있어야 함

### 10.2 성능

- 앱 초기 진입 후 첫 척도 선택까지 3초 이내가 목표
- 척도 데이터는 가능하면 번들 포함 방식으로 로딩 실패를 줄인다

### 10.3 안정성

- 인터넷이 없어도 척도 렌더링과 로컬 저장은 가능해야 한다
- 시트 전송 실패 시 재전송이 가능해야 한다

### 10.4 보안 및 개인정보

- 이름 전체, 주민번호, 상세 주소 등 과도한 개인정보는 기본 수집하지 않는다
- URL 기반 전송을 사용할 경우 내부 운영용으로 제한한다
- 가능하면 토큰 검증 또는 조직 내 URL 관리 정책을 병행한다

## 11. 데이터 구조 원칙

### 11.1 검사 기록

검사 1건은 `record_id` 기준으로 관리한다.

포함 항목 예시:

- 검사일
- 담당자
- 대상자 식별 메모
- 척도 ID와 척도명
- 총점
- 결과 밴드
- 응답자 요약
- 현장 메모
- 원본 JSON

### 11.2 문항 응답

문항 응답은 `척도문항응답` 시트에 행 단위로 저장한다.

포함 항목 예시:

- record_id
- 문항 ID
- 문항 번호
- 문항 텍스트
- 선택 응답
- 점수
- 하위문항 여부

### 11.3 척도 마스터

척도 정의는 아래 세 시트로 분리 저장한다.

- `척도마스터`
- `척도문항마스터`
- `척도선택지마스터`

이 구조를 사용하면 척도 추가, 검증, 운영 중 비교가 쉬워진다.

### 12.1 메인 화면

- 구글 시트 연동 설정

### 12.2 검사 화면

- 응답자 정보 영역

- 결과보기 버튼

### 12.3 결과 화면

- 총점
- 결과 밴드
- 설명 문구
- 문항별 응답 요약

## 13. 성공 지표

### 13.1 사용성 지표

- 실무자가 별도 교육 없이 첫 검사 완료까지 5분 이내
- 척도 선택 후 결과 산출까지 평균 3분 이내
- 현장 사용 중 조작 오류 보고 건수 감소

### 13.2 운영 지표

- 검사 후 시트 반영 성공률 95% 이상
- 누락 없는 척도 마스터 적재
- 수기 점수 계산 건수 80% 이상 감소

  - 대응: Web App 배포 설정 표준화, 상태 확인 절차 마련

- 개인정보 과수집 위험
  - 대응: 입력 필드 최소화, 운영 지침 문서 병행

## 16. 운영 정책 제안

- 현장 실무자는 웹앱만 사용한다.
- 관리자는 구글 시트에서 누적 기록을 확인한다.
- AppSheet는 필요 시 조회 전용 또는 관리자용 보조 화면으로만 사용한다.
- 척도 원문과 결과 해석은 운영 책임자가 최종 검수한다.

## 17. 오픈 이슈

- 기관에서 반드시 필요한 척도 추가 목록 확정 필요
- 결과 문구의 기관 표준안 확정 필요
- 익명 운영과 로그인 운영 중 보안 정책 확정 필요
- 장기적으로 대상자 재검 비교 기능이 필요한지 검토 필요

## 18. 최종 제안

본 프로젝트의 1차 제품은 `현장용 전용 웹앱`으로 추진하는 것이 적절하다.  
정신건강 사회복지사의 실제 사용 흐름은 `빠른 척도 실행`이 가장 중요하며, 이 목적에는 AppSheet보다 전용 웹앱이 더 적합하다.

다만 운영과 누적 관리 측면에서 구글 시트는 매우 유용하므로, 최적 구조는 아래와 같다.

- 입력과 검사 수행: 웹앱
- 누적 저장과 운영 관리: 구글 시트
- 조회 보조 화면: 필요 시 AppSheet
