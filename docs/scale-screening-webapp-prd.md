# MindMap 다시서기 척도관리 웹앱 PRD

문서 상태: Draft v3  
작성일: 2026-03-26  
대상 프로젝트: `google-apps-script-scale-webapp`  
참조 프로젝트: `scale-screening-web-app`, `google-apps-script-target-sheet`

## 1. 제품 요약

`MindMap 다시서기 척도관리 웹앱`은 다시서기종합지원센터 정신건강팀이 현장에서 정신건강 척도검사를 빠르게 수행하고, 결과를 즉시 확인하고, 검사 기록을 구글 시트에 누적 저장하며, 대상자별 반복 검사 결과를 검색·비교·분석할 수 있도록 만든 내부용 웹앱이다.

현재 제품은 `Google Apps Script 웹앱 + Google Sheets` 구조로 운영되며, 별도 서버 없이 Google 계정 기반 브라우저 환경에서 접속한다. 실무자는 로그인 후 척도를 선택해 검사를 수행하고, 필요 시 구글 시트에서 자신의 저장 결과와 대상자별 결과 추이를 함께 확인한다. 관리자는 계정과 가입신청, 저장 결과 흐름, 분석용 시트 구성을 함께 관리한다.

## 2. 배경과 문제 정의

기존 척도 운영은 기관 홈페이지, 종이 문항지, 수기 기록, 개별 엑셀, 구글 폼 등으로 분산되기 쉽다. 이 방식은 아래 문제를 만든다.

- 현장에서 필요한 척도를 찾는 시간이 길다.
- 점수 계산과 결과 정리가 수작업에 의존한다.
- 상담자마다 기록 항목과 표현이 달라 누적 관리가 어렵다.
- 결과 저장 위치가 분산되어 재열람과 비교가 어렵다.
- 동일 대상자의 반복 검사 결과를 날짜별로 비교하기 어렵다.
- 고위험군이나 위기개입 대상자의 변화 추이를 한눈에 보기 어렵다.
- 모바일/태블릿 환경에서 대상자 앞에서 사용하기 불편하다.
- 관리자 입장에서 사용자 계정과 기록 권한을 통제하기 어렵다.

## 3. 제품 목표

### 3.1 핵심 목표

- 실무자가 1분 이내에 원하는 척도를 열고 검사를 시작할 수 있어야 한다.
- 검사 완료 즉시 점수와 결과 등급이 자동 계산되어야 한다.
- 검사 결과를 인쇄 가능한 결과지로 바로 제시할 수 있어야 한다.
- 저장 결과를 구글 시트에 누적 반영할 수 있어야 한다.
- 실무자가 대상자 이름을 검색해 검사 날짜별 점수 변화를 비교할 수 있어야 한다.
- 구글 시트 대시보드에서 척도별 점수 변화 그래프를 확인할 수 있어야 한다.
- 내부 로그인, 가입신청, 관리자 계정관리를 앱 안에서 처리할 수 있어야 한다.
- 척도를 데이터 기반으로 관리해 신규 척도 추가가 쉬워야 한다.

### 3.2 비목표

- 의료 진단 시스템 구축
- EMR 연동
- 외부 대중 서비스 운영
- 주민등록번호, 상세 주소 등 고위험 개인정보 수집
- 복잡한 조직 결재 워크플로우 구축

## 4. 제품 방향

현재 제품 방향은 `현장용 검사 수행 도구 + 내부 기록 관리/분석 도구`의 결합형이다.

- 웹앱: 로그인, 척도 선택, 문항 응답, 자동 채점, 결과 확인, 인쇄, 저장
- 구글 시트: 실무자와 관리자 모두가 사용하는 누적 기록 저장, 문항 응답 저장, 척도 마스터 저장, 대상자 검색, 결과 비교, 고위험군 조회, 대시보드 분석
- 관리자 화면: 계정 관리, 가입신청 처리, 기록 운영 상태 확인

초기 로컬 프로토타입은 `scale-screening-web-app`에 남겨 두고, 실제 운영본은 `google-apps-script-scale-webapp`과 `google-apps-script-target-sheet`를 기준으로 발전시킨다.

## 5. 대상 사용자

### 5.1 1차 사용자

- 다시서기종합지원센터 정신건강팀 실무자
- 현장상담, 사례관리, 초기면접 담당자
- 위기개입 및 정신건강 상담 담당자

### 5.2 2차 사용자

- 팀장 또는 관리자
- 계정 승인 및 운영 담당자
- 기록 취합 및 실적 점검 담당자

## 6. 대표 사용 시나리오

### 6.1 현장 검사

실무자가 태블릿이나 노트북에서 앱에 로그인하고, PHQ-9, GAD-7, CRI 같은 척도를 선택해 검사한다. 대상자 정보와 검사 정보를 입력한 뒤 문항에 응답하면 결과 요약이 즉시 계산되고, 필요 시 A4 한 장 결과지로 출력한다.

### 6.2 위기상황 평가

상담자가 CRI를 선택해 자타해 위험, 정신상태, 기능수준, 지지체계를 평가하고, A~E 등급 결과를 즉시 확인한다. 결과에 따라 현장 대응 수준을 빠르게 판단한다.

### 6.3 기록 누적

검사 후 저장 버튼을 눌러 결과를 시트에 누적 반영한다. 실무자는 시트에서 자신의 저장 결과와 문항응답 누적 기록을 다시 확인하고, 관리자는 전체 기록과 운영 상황을 점검한다.

### 6.4 대상자 비교 분석

실무자 또는 관리자가 구글 시트의 `척도대시보드` 시트에서 대상자 이름을 검색한다. 선택된 대상자의 검사 기록이 날짜순으로 정리되고, 각 척도별 점수 변화가 선 그래프로 표시된다. 동명이인 구분이 필요할 때는 생년월일을 보조 필터로 입력한다.

### 6.5 계정 운영

신규 실무자는 가입신청 페이지에서 계정을 신청한다. 가입신청은 `승인 대기` 상태로 접수되며, 관리자는 관리자 화면에서 계정 승인, 활성/비활성, 수정, 삭제를 수행한다.

## 7. 현재 제공 범위

현재 운영본 기준 포함 범위는 아래와 같다.

- Apps Script 웹앱 배포
- 내부 로그인
- 별도 가입신청 페이지
- 관리자 계정관리
- 최근 저장 결과 조회
- 구글 시트 동기화
- 실무자용 구글 시트 기록 확인
- 문항응답 저장
- 척도 마스터 저장
- 대상자 검색 대시보드
- 척도별 점수 변화 그래프
- 고위험군 조회 시트
- 인쇄용 결과지
- 서명 입력
- 결과 저장 후 재열람

## 8. 제공 척도 범위

현재 제공 척도는 총 9개다.

- 우울 `PHQ-9`
- 불안 `GAD-7`
- 조기정신증 `mKPQ-16`
- 조울증 `K-MDQ`
- 스트레스 `PSS-10`
- 불면 `ISI-K`
- 알코올중독 `AUDIT-K`
- 사건충격척도 `IES-R`
- 정신건강위기평정척도 `CRI`

향후 척도 추가는 원칙적으로 `질문 데이터 + 채점 규칙 데이터`를 넣는 방식으로 처리한다. 단, CRI처럼 별도 등급 로직이 필요한 척도는 전용 평가 함수를 추가한다.

## 9. 기능 요구사항

### FR-01 로그인

- 사용자는 앱 접속 시 로그인 화면을 먼저 보아야 한다.
- 로그인 성공 후에만 검사 화면에 접근할 수 있어야 한다.
- 세션이 만료되면 다시 로그인 화면으로 돌아가야 한다.

### FR-02 가입신청

- 로그인 화면에서 가입신청 버튼으로 별도 가입 페이지에 이동할 수 있어야 한다.
- 가입신청 입력 항목은 이름, 아이디, 비밀번호, 비밀번호 확인으로 제한한다.
- 소속은 `다시서기`로 고정한다.
- 가입신청은 승인 대기 상태로 저장되고, 관리자 승인 후 로그인 가능해야 한다.

### FR-03 관리자 기능

- 관리자 계정은 계정 목록을 조회할 수 있어야 한다.
- 관리자는 계정 생성, 수정, 활성/비활성, 삭제를 수행할 수 있어야 한다.
- 관리자는 가입신청 내역을 조회하고 승인 또는 거절할 수 있어야 한다.

### FR-04 척도 선택

- 로그인 후 사용자는 척도 목록을 즉시 볼 수 있어야 한다.
- 각 척도는 이름, 약어, 권장 연령을 표시해야 한다.
- 척도 목록은 모바일과 태블릿에서 터치하기 쉬워야 한다.

### FR-05 검사 정보 입력

- 검사일은 필수 입력이다.
- 담당자, 대상자, 생년월일, 비고를 입력할 수 있어야 한다.
- 척도별 응답자 정보 문항이 있으면 같이 렌더링해야 한다.

### FR-06 문항 응답

- 각 문항은 선택형으로 응답 가능해야 한다.
- 필수 응답 누락 시 어떤 항목이 비었는지 안내해야 한다.
- 응답 진행률을 퍼센트와 항목 수로 보여줘야 한다.
- 긴 문항도 모바일에서 읽기 쉬워야 한다.
- CRI는 `자타해 위험`, `정신상태`, `기능수준`, `지지체계` 구간이 화면에서 구분되어야 한다.

### FR-07 결과 계산

- 척도별 점수 계산 규칙을 자동 적용해야 한다.
- 결과 요약에는 총점, 구간 또는 등급, 설명 문구가 포함되어야 한다.
- CRI는 총점 외에 A~E 등급과 하위영역 점수를 함께 제시해야 한다.
- 결과 화면에는 문항별 응답 요약이 포함되어야 한다.

### FR-08 결과 저장

- 사용자는 계산한 결과를 저장할 수 있어야 한다.
- 저장 시 검사기록, 문항응답, 척도 마스터가 필요한 구조로 시트에 반영되어야 한다.
- 최근 저장 결과를 앱 안에서 다시 조회할 수 있어야 한다.

### FR-09 인쇄와 서명

- 결과지는 A4 한 장 기준으로 출력할 수 있어야 한다.
- 결과지 하단에 담당자 수기 서명 영역이 있어야 한다.
- 결과지 상단에는 `다시서기종합지원센터 / 정신건강팀`이 고정 표기되어야 한다.

### FR-10 구글 시트 동기화

- 대상 시트를 지정할 수 있어야 한다.
- 시트 준비, 척도 마스터 동기화, 결과 저장이 가능해야 한다.
- 실무자도 자신의 검사 결과를 시트에서 조회하고 업무상 활용할 수 있어야 한다.
- 관리자는 전체 누적 기록을 시트에서 점검하고 운영 관리에 활용할 수 있어야 한다.
- 대상자 이름으로 검사 기록을 검색할 수 있는 대시보드 시트가 있어야 한다.
- 동일 대상자의 검사 날짜별 점수 변화를 척도별 선 그래프로 볼 수 있어야 한다.
- 고위험 결과를 별도로 모아보는 조회 시트가 있어야 한다.
- 시트에는 아래 탭 구조를 유지한다.
  - `척도검사기록`
  - `척도문항응답`
  - `실무자보기`
  - `고위험군보기`
  - `척도대시보드`
  - `척도설정`
  - `척도마스터`
  - `척도문항마스터`
  - `척도선택지마스터`

## 10. 비기능 요구사항

### 10.1 사용성

- 모바일 우선 설계
- 태블릿 가로/세로 대응
- 큰 버튼과 명확한 선택 상태
- 50대 이상 실무자도 큰 교육 없이 사용할 수 있어야 함

### 10.2 성능

- 로그인 이후 첫 화면 로딩 체감 시간이 짧아야 한다.
- 척도 데이터는 번들 방식으로 제공해 로딩 실패를 줄여야 한다.
- 시트 접근 횟수는 최소화하고, 가능한 한 메모리에서 처리 후 일괄 저장해야 한다.
- 구글 시트 대시보드는 원본 저장 시트와 분리된 조회 시트 기반으로 구성해 체감 로딩 속도를 유지해야 한다.

### 10.3 안정성

- 시트 연동 실패 시 사용자에게 명확한 오류 메시지를 제공해야 한다.
- 세션 만료, 권한 부족, 중복 아이디 상황을 명확히 안내해야 한다.
- 관리자 작업 중 데이터 충돌을 방지해야 한다.

### 10.4 보안 및 개인정보

- 과도한 개인정보 수집을 피한다.
- 계정과 세션 정보는 Apps Script `PropertiesService`와 세션 구조를 통해 관리한다.
- 비밀번호는 `salt + 반복 SHA-256 + pepper` 방식으로 저장하고, `pepper`는 스크립트 속성에 자동 초기화한다.
- 관리자 기능은 관리자 역할 계정에서만 접근 가능해야 한다.
- 외부 공개보다는 내부 운영을 우선한다.

## 11. 데이터 구조 원칙

### 11.1 검사 기록

검사 1건은 `record_id` 기준으로 관리한다.

포함 항목 예시:

- 검사일
- 담당자
- 대상자
- 생년월일
- 비고
- 척도 ID와 척도명
- 총점
- 결과 구간 또는 등급
- 응답 진행률
- 저장시각
- 원본 JSON

### 11.2 문항 응답

문항 응답은 `척도문항응답` 시트에 행 단위로 저장한다.

포함 항목 예시:

- record_id
- question_id
- 문항 번호
- 문항 텍스트
- 응답값
- 점수
- 하위문항 여부

### 11.3 실무자 조회 시트

`실무자보기` 시트는 `척도검사기록` 원본을 바탕으로 실무자가 바로 검색·정렬·필터할 수 있는 조회용 시트다.

포함 항목 예시:

- 검사일
- 대상자
- 생년월일
- 척도
- 점수값
- 점수표시
- 결과구간
- 담당자
- 비고
- 경고여부
- 기록고유값

### 11.4 고위험군 조회 시트

`고위험군보기` 시트는 원본 기록 중 경고 플래그가 있거나 CRI 등급이 위험 수준인 사례를 모아보는 조회용 시트다.

포함 항목 예시:

- 검사일
- 대상자
- 생년월일
- 척도
- 점수표시
- 결과구간
- 담당자
- 경고내용
- 기록고유값

### 11.5 대시보드 시트

`척도대시보드` 시트는 대상자 이름과 생년월일을 기준으로 검사 이력을 검색하고, 날짜별 척도 점수 변화 그래프를 보여주는 분석 시트다.

포함 요소 예시:

- 대상자명 입력 셀
- 생년월일 필터 셀
- 대상자 검사 이력 표
- 척도별 점수 변화 피벗 범위
- 선 그래프 차트

### 11.6 설정 시트

`척도설정` 시트는 조회 시트 구조와 사용법, 연결 시트 이름 등 운영 설정을 설명하는 안내 시트다.

포함 항목 예시:

- 기록 시트 이름
- 문항응답 시트 이름
- 실무자 보기 시트 이름
- 고위험군 보기 시트 이름
- 대시보드 시트 이름
- 검색 사용법

### 11.7 계정 데이터

계정 데이터는 시트가 아니라 `PropertiesService` 기반 저장 구조를 사용한다.

포함 항목 예시:

- username
- displayName
- role
- active
- affiliation
- notes
- createdAt
- updatedAt

### 11.8 가입신청 데이터

가입신청은 별도 요청 배열로 관리한다.

포함 항목 예시:

- requestId
- username
- displayName
- status
- reviewedBy
- reviewedAt
- linkedUsername

## 12. 화면 구성 요구사항

### 12.1 로그인 화면

- 좌측 정보 패널
- 우측 로그인 카드
- 가입신청 버튼

### 12.2 가입신청 화면

- 앱 설명 없이 가입신청 카드만 표시
- 이름, 아이디, 비밀번호, 비밀번호 확인 입력
- 로그인으로 돌아가기 버튼

### 12.3 메인 검사 화면

- 척도 선택
- 검사정보 입력
- 응답자 정보
- 문항응답
- 결과 계산 버튼

### 12.4 결과 화면

- 총점
- 결과 구간 또는 등급
- 요약 설명
- 검사정보 요약
- 문항별 응답 내역
- 저장 버튼
- 서명 영역

### 12.5 관리자 화면

- 계정 수 / 신청 건수 요약
- 계정 등록 및 수정 폼
- 계정 목록 테이블
- 가입신청 목록 테이블

### 12.6 구글 시트 대시보드

- `대상자명` 검색 입력 셀
- `생년월일` 보조 필터 입력 셀
- 날짜순 검사 이력 표
- 척도별 점수 변화 그래프
- 고위험군 조회 시트와 연결 가능한 구조

## 13. 성공 지표

### 13.1 사용성 지표

- 로그인 후 첫 검사 시작까지 1분 이내
- 척도 선택 후 결과 산출까지 평균 3분 이내
- 현장 사용 중 수기 계산 의존도 대폭 감소

### 13.2 운영 지표

- 시트 반영 성공률 95% 이상
- 계정 생성/비활성화 처리 오류 최소화
- 척도 추가 시 데이터 파일 중심으로 반영 가능
- 대상자 검색 후 그래프 확인까지 10초 이내

## 15. 리스크와 대응

- Apps Script 캐시로 인해 배포 반영이 늦게 보일 수 있음
  - 대응: 배포 버전 관리, 새 배포, 강력 새로고침 가이드
- 로그인/세션 처리 속도가 느릴 수 있음
  - 대응: CacheService, 상태 조회 1회화, 최근 기록 경량 조회, 일괄 시트 접근
- 관리자와 일반 사용자 동시 접근 시 충돌 가능성
  - 대응: LockService 사용
- 척도 해석 문구의 기관 내 합의 필요
  - 대응: 척도별 결과 문구를 데이터 설정으로 분리
- 대시보드 수식이 복잡해질 경우 시트 로딩이 느려질 수 있음
  - 대응: 원본 시트와 조회 시트를 분리하고, 조회 시트는 수식 기반, 원본 저장은 배치 저장으로 유지

## 16. 운영 정책 제안

- 실무자는 웹앱을 주 도구로 사용하되, 저장 결과 확인, 대상자 비교, 점수 변화 분석을 위해 구글 시트도 함께 사용한다.
- 관리자는 웹앱 관리자 화면과 구글 시트를 함께 사용한다.
- 척도 원문과 해석 문구는 운영 책임자가 최종 검수한다.
- 신규 척도 추가 시 데이터 파일, 채점 규칙, 결과 문구를 함께 검수한다.

## 17. 오픈 이슈

- CRI 외 추가 위기척도 도입 여부
- 가입신청 승인 기준과 처리 SLA 정리 필요
- 관리자 계정 초기값 운영 정책 정리 필요
- 대상자 반복검사 비교 기능의 시각화 범위 추가 검토 필요
- 성능 최적화 이후 목표 응답시간 재측정 필요

## 18. 최종 제안

현재 프로젝트는 `Apps Script 웹앱 + 구글 시트 저장/분석` 구조가 가장 현실적이다.

이 구조는 다음 장점을 가진다.

- 별도 서버 없이 내부 운영 가능
- 구글 시트를 실무자와 관리자가 함께 활용할 수 있음
- 현장용 화면을 자유롭게 설계 가능
- 관리자 기능을 앱 내부에 통합 가능
- 반복 검사 결과를 구글 시트 대시보드에서 비교·분석 가능

따라서 단기적으로는 `현재 Apps Script 웹앱을 안정화하고 성능 최적화하는 것`, 중기적으로는 `구글 시트 대시보드 기반 기록 조회와 비교 분석 기능을 강화하는 것`이 가장 적절한 방향이다.
