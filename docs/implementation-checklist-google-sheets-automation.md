## 1. 구현 기본 결정

이번 구현은 아래 기본 결정을 기준으로 진행한다.

- **운영 방식:** 방식 A(집계값 입력)를 기본으로 하고, 기존 케이스 DB를 병행 참조하는 혼합형 MVP로 시작한다.
- **입력 UX:** 담당자는 `일일활동DB`에만 직접 입력한다.
- **집계 방식:** 보고·월별뷰 시트는 수식 기반으로 우선 구축하고, Apps Script는 입력 보조와 운영 자동화에만 사용한다.
- **구조 선택:** 사용자 입력용 `일일활동DB`는 가로형(wide) 구조로 두고, 필요 시 숨김 시트 `일일활동정규화`를 두어 분석용 세로형(long) 구조를 보조한다.
- **월별뷰 구현:** 월별 시트를 복제하지 않고, `일일실적_월별뷰` 1개 시트에서 연도/월 파라미터로 표시 월을 바꾸는 방식을 기본안으로 채택한다.

## 3. 작업 단계별 체크리스트

### 3.1 착수 준비

- [ ] 구글시트 마스터 파일 1개를 생성한다.
- [ ] 파일 소유자 계정과 운영 관리자 계정을 확정한다.
- [ ] 스프레드시트 시간대를 `Asia/Seoul`로 설정한다.
- [ ] 기존 엑셀 원본의 최신본과 기준 연도(예: 2026)를 확정한다.
- [ ] 담당자 목록 8명의 현재 재직 여부를 검증한다.
- [ ] 기존 보고 양식 중 반드시 유지해야 하는 레이아웃을 캡처해 둔다.

완료 기준:

- 마스터 파일이 생성되어 있고, 관리자 1명 이상이 편집 권한을 가진다.
- 기준 연도와 담당자 목록이 확정되었다.

### 3.2 기준/설정 시트 구축

- [ ] `기준` 시트를 만든다.
- [ ] 담당자 목록 범위를 만든다.
- [ ] 항목 목록(상담/조치/사례관리)을 만든다.
- [ ] 보고 기준 연도 셀과 기본 월 셀을 만든다.
- [ ] 이름정의 범위(`workers`, `cfg_report_year` 등)를 등록한다.
- [ ] `도움말` 시트를 만들어 입력 원칙과 예시를 적는다.

완료 기준:

- `일일활동DB`의 드롭다운과 보고 시트 수식에서 `기준` 시트를 참조할 수 있다.
- 관리자 외에는 `기준` 시트를 수정할 수 없도록 보호되어 있다.

### 3.3 `일일활동DB` 구축

- [ ] `일일활동DB` 시트를 생성한다.
- [ ] A:U 열 구조를 PDR 기준으로 배치한다.
- [ ] 날짜 열에 날짜만 허용하는 유효성 검사를 적용한다.
- [ ] 담당자 열에 드롭다운을 적용한다.
- [ ] 건수 열(C:S)에 0 이상 정수만 허용하는 유효성 검사를 적용한다.
- [ ] T/U 열은 입력자/입력일시 전용 자동 기록 열로 정의한다.
- [ ] 헤더 고정, 필터, 조건부 서식(중복 입력 경고)을 적용한다.
- [ ] 샘플 5건을 입력해 유효성 오류와 표시 형식을 검증한다.

완료 기준:

- 담당자가 날짜, 담당자, 건수만 입력할 수 있다.
- 잘못된 날짜/음수/목록 외 담당자 입력이 차단된다.
- 같은 날짜·담당자 조합의 중복 여부를 시각적으로 식별할 수 있다.

### 3.4 기존 케이스 DB 정비

- [ ] `입원현황`, `시설입소현황`, `연계대장`, `조치후 사례관리`, `상담현황(2026)` 시트의 실제 열 위치를 확정한다.
- [ ] 월, 날짜, 담당자, 유형 열의 위치를 문서화한다.
- [ ] 누락 필드가 있으면 보조 열 또는 표준화 규칙을 정한다.
- [ ] 월 값이 숫자/텍스트 혼재인지 확인하고 필요 시 표준화한다.
- [ ] 날짜 값이 텍스트로 들어간 행이 있는지 점검한다.

완료 기준:

- 집계 수식에서 참조할 열이 모두 확정되어 있다.
- 월/날짜/담당자 형식 불일치 규칙이 문서화되어 있다.

### 3.5 `일일실적_월별뷰` 구축

- [ ] `일일실적_월별뷰` 시트를 생성한다.
- [ ] 상단에 `기준연도`, `기준월` 파라미터 셀을 둔다.
- [ ] 기존 월별 일일실적의 레이아웃을 최대한 유지한다.
- [ ] 날짜 블록(담당자 8행 + 일일누계 + 총누계)을 수식 기반으로 채운다.
- [ ] 일일누계/총누계는 직접 입력이 아닌 합계 수식으로 계산한다.
- [ ] 표시 월을 바꾸었을 때 28/29/30/31일이 올바르게 반영되는지 검증한다.

완료 기준:

- 월 복사 없이 파라미터 변경만으로 다른 월을 조회할 수 있다.
- 기존 양식과 유사한 시각 구조를 유지한다.

### 3.6 보고 시트 재구성

- [ ] `법인월보고용` 수식을 날짜 기반으로 다시 작성한다.
- [ ] `분기보고용` 수식을 날짜/월 범위 기반으로 다시 작성한다.
- [ ] `입원실적`을 `입원현황` 기반 `COUNTIFS`로 재작성한다.
- [ ] `시설입소실적`을 `시설입소현황` 기반으로 재작성한다.
- [ ] `연계실적`을 `연계대장` 기반으로 재작성한다.
- [ ] `조치후사례관리(집계)`를 `조치후 사례관리` 기반으로 재작성한다.
- [ ] `#REF!`가 더 이상 발생하지 않도록 시트명 의존 수식을 제거한다.

완료 기준:

- 모든 집계 시트가 DB 시트만 참조한다.
- 기준 연도 변경 시 주요 보고 값이 자동으로 갱신된다.

### 3.7 Apps Script 구축

- [ ] `onOpen` 메뉴를 만든다.
- [ ] `onEdit`로 `일일활동DB`의 입력자/입력일시를 자동 기록한다.
- [ ] 날짜 텍스트 입력 감지 및 보정 로직을 넣는다.
- [ ] 중복 입력 감지 시 노트 또는 경고 메시지를 표시한다.
- [ ] 관리자용 검증 메뉴(`검증 실행`, `월별뷰 재계산`)를 추가한다.
- [ ] 선택 기능으로 월말 PDF 메일 발송 함수를 만든다.
- [ ] 오류 로그 기록 방식을 정한다. 숨김 시트 또는 Apps Script 로그 중 택1

완료 기준:

- 사용자 편집만으로 타임스탬프와 기본 검증이 수행된다.
- 관리자가 메뉴에서 점검 작업을 직접 실행할 수 있다.

### 3.8 데이터 이관 및 비교 검증

- [ ] 최근 1개월 데이터를 샘플로 이관한다.
- [ ] `일일활동DB` 합계와 기존 월별 시트 합계를 비교한다.
- [ ] `입원현황`과 `입원실적` 합계를 비교한다.
- [ ] `연계대장`과 `연계실적` 합계를 비교한다.
- [ ] 차이 발생 시 원인 분류 표를 작성한다. 수식 문제 / 원본 데이터 누락 / 기준 정의 차이
- [ ] 최소 2주간 병행 운영한다.

완료 기준:

- 핵심 보고 항목의 차이가 허용 범위 내이거나 원인이 모두 설명 가능하다.
- 병행 운영 기간 동안 치명적 오류가 없다.

### 3.9 운영 전환

- [ ] 집계 시트 보호를 활성화한다.
- [ ] 팀원 교육 1회를 진행한다.
- [ ] 입력 가이드 1페이지를 공유한다.
- [ ] 실무자가 직접 1일 입력을 시연한다.
- [ ] 장애 대응 담당자와 연락 채널을 정한다.

완료 기준:

- 팀원 전원이 입력 위치를 알고 있다.
- 운영 시작일 이후 집계 시트 직접 수정이 발생하지 않는다.

## 4. 테스트 시나리오

### 필수 테스트

| TC | 시나리오 | 기대 결과 |
| --- | --- | --- |
| TC-01 | 날짜를 `2026-03-16`으로 입력 | 정상 저장 |
| TC-02 | 날짜를 `3/16` 텍스트로 입력 | 날짜로 보정되거나 경고 표시 |
| TC-03 | 담당자에 목록 외 값 입력 | 저장 차단 |
| TC-04 | 건수에 `-1` 입력 | 저장 차단 |
| TC-05 | 동일 날짜/담당자 2행 입력 | 중복 경고 표시, 집계는 합산 |
| TC-06 | `기준연도`를 2026 → 2027로 변경 | 보고 시트 값이 연도 기준으로 갱신 |
| TC-07 | `기준월`을 2월로 변경 | 월별뷰가 2월 일수 기준으로 표시 |
| TC-08 | `입원현황`에 자의입원 1건 추가 | `입원실적` 반영 |

### 승인 기준

- 필수 테스트 8건이 모두 통과한다.
- `#REF!`와 같은 구조적 오류가 남아 있지 않다.

## 5. 운영 리스크와 대응

| 리스크 | 영향 | 대응 |
| --- | --- | --- |
| 담당자 목록 변경 | 드롭다운 오류 | `기준` 시트만 수정하도록 운영 규칙화 |
| 수식 범위 실수 | 보고 값 왜곡 | 집계 시트 보호 + 검증 메뉴 제공 |
| 케이스 DB 형식 불일치 | 집계 누락 | 열 매핑표 유지, 월 1회 점검 |
| 대량 데이터로 인한 지연 | 사용자 불편 | 열 전체 참조 최소화, 필요 시 보조 집계 시트 사용 |
| Apps Script 권한 문제 | 자동화 일부 실패 | Workspace 계정 확인, 수동 메뉴 대체 절차 마련 |

## 6. 완료 정의 (Definition of Done)

아래 조건을 모두 만족하면 1차 구축 완료로 본다.

- `일일활동DB` 입력이 운영 가능하다.
- `일일실적_월별뷰`가 파라미터 기반으로 작동한다.
- `법인월보고용`, `분기보고용`, `입원실적`, `시설입소실적`, `연계실적`이 DB 참조 기반으로 재구성되어 있다.
- 입력자/입력일시 자동 기록이 동작한다.
- 2주 병행 운영에서 치명적 데이터 불일치가 없다.
